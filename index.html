The interdependency problem

ERP risks rarely occur in isolation — they amplify each other. Understanding the cascade is essential.

Cascade example 1

Poor training (People) → users avoid the system → business units build shadow spreadsheets → data inconsistency → management cannot trust reports (Technology).

Cascade example 2

Shifting requirements (Process) → late customisation requests → integration timelines slip → go-live delayed → executive sponsor loses confidence (People) → budget cut.

Cascade example 3

Integration failure (Technology) → manual workarounds → staff overloaded → change resistance grows (People) → adoption never reaches target level.

People dimension

People risks — deep analysis

People risks are the most underestimated and the most common cause of ERP failure. Technology rarely fails alone — it fails when people don't adopt, don't know how to use it, or don't support it.

Change management maturity model

Where is your organisation on the OCM spectrum? Each level requires different interventions.

Level 1 — Unaware

Critical risk

No OCM programme. Leadership assumes users will "just adapt." No communication plan, no change champions, no training beyond IT sessions. Intervention needed immediately: appoint OCM lead, conduct stakeholder impact assessment.

Level 2 — Aware

High risk

OCM recognised as important but under-resourced. Some communication happening, training planned but not yet designed. Intervention: formalise OCM workstream, assign budget, build change champion network.

Level 3 — Developing

Medium risk

OCM plan exists. Change champions identified. Training curriculum designed but not yet delivered. Intervention: run stakeholder readiness surveys, iterate training based on feedback, increase exec visibility.

Level 4 — Embedded

Low risk

OCM fully integrated into the project plan. Readiness measured monthly. Training delivered, adoption tracked post go-live. Change champions active. Executive sponsor visible and communicating.

Process dimension

Process risks — deep analysis

Process risks often originate in the gap between how the business thinks it works and how it actually works — and between what the ERP can do out of the box and what the business demands it does.

The customisation decision framework

Before approving any customisation, every request should pass through this four-question gate.

Q1 — Is it legally or regulatorily required?

If yes: customise, document, flag for every future upgrade cycle. If no: proceed to Q2.

Q2 — Does it create competitive differentiation?

If yes and quantifiable: escalate to steering committee with cost-benefit analysis. If no: proceed to Q3.

Q3 — Can we change the process to avoid it?

This is the right question to ask. 80% of customisation requests should end here with a process change. Document the process change and train users.

Q4 — What is the full lifetime cost?

Build + test + document + re-test at each upgrade + maintain. If the cost over 5 years exceeds benefit: reject. Only approve if all other options are exhausted.

Data migration — the six-stage discipline

Data migration is the most technically underestimated workstream. Each stage must be completed before the next begins.

1
Inventory & discovery

Document all source systems, data objects, volumes, owners. Identify records that must migrate vs. archive vs. discard.

2
Data quality assessment

Profile source data for completeness, accuracy, duplicates, and consistency. Assign quality score per data object. Prioritise remediation.

3
Cleansing & enrichment

Business data owners — not IT — cleanse records. IT builds the tooling; business owns the data quality. Document all transformation rules.

4
Mock migration × 3

Run at least three full mock migrations to production-equivalent environment. Measure success rate against acceptance thresholds. Fix and re-run.

5
Reconciliation & sign-off

Business owners formally validate migrated data in the new system. Define minimum acceptable completeness thresholds (e.g. ≥98% for financial records).

6
Cutover & hypercare

Execute cutover migration with a defined rollback trigger. Maintain data reconciliation reports during the first 30-day hypercare period.

Technology dimension

Technology risks — deep analysis

Technology risks are often the most visible but can be the hardest to reverse once the system is live. Integration complexity and infrastructure sizing are the two areas most frequently underestimated at project start.

Integration architecture principles

A stable integration architecture prevents the most common technology failure mode — silent data discrepancies between systems.

Use middleware / iPaaS

Point-to-point integrations create a web of brittle connections. An integration layer (MuleSoft, Boomi, Azure Integration Services) centralises error handling, logging, and reprocessing.

Define an integration inventory early

By week 4 of the project: list every system, data flow direction, frequency, volume, and owner. Undiscovered integrations found at go-live are the number one cause of emergency delays.

Build error alerting from day one

Every integration must have monitoring. Silent failures — where data simply stops flowing — are far more dangerous than noisy failures. Instrument every interface with alerts.

Plan for regression testing

After every ERP patch or upgrade, all integrations must be regression tested. Build an automated integration test suite before go-live — retrofitting it post go-live almost never happens.

Security configuration checklist

Security misconfiguration is the most common compliance risk in ERP go-lives. These seven controls should be verified before cutover.

Role-based access control aligned to job function, not seniority — verified by HR and line managers
Segregation of duties enforced — no single user can create, approve, and pay a transaction
Audit logging enabled for all financial and HR data transactions from day one
All default vendor passwords and demo accounts removed or disabled
Data residency and encryption requirements confirmed (particularly for cloud/SaaS deployments)
Independent security penetration test completed and findings remediated before go-live
Disaster recovery and backup restore tested — not just scheduled, actually executed and timed

Mitigation timeline

When to act on each risk — mapped to typical ERP project phases.

Typical ERP phases

Prepare Weeks 1–4 — Scope, team, governance
Explore Weeks 5–12 — Fit-gap, process design
Realise Weeks 13–30 — Build, configure, integrate
Deploy Weeks 31–36 — UAT, cutover, go-live
Run Post go-live — Hypercare, stabilisation

Key principle

The cost of fixing a risk doubles at each phase transition. A requirements gap found in Explore costs 1×. Found in Realise: 4×. Found in Deploy: 10×. Found post go-live: 20×.

Front-load your risk mitigation. Every risk identified in the Prepare phase saves significant rework, cost, and credibility later in the project.
